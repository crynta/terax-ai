# Terax SOTA Plan (2026-06-11)

Supersedes `docs/architecture-review-2026-06.md` and `docs/improvement-plan-2026-06.md` (both written 2026-06-10, one day and six commits earlier). This document re-audits the codebase after the webview-native Pi agent landed, states the target architecture for the best version of this idea, and lays out the execution plan. Three parallel deep audits (pi runtime, workflow, Rust backend delta) plus direct verification back every claim.

---

## 1. The idea, stated precisely

Terax is an ultra-lightweight, AI-native terminal emulator. The product thesis that everything else must serve:

- **Terminal first.** xterm + PTY is the primary surface; editor, explorer, git are supporting cast.
- **Agent as a primitive, not a bolt-on.** A local agent runtime with sessions, fork/rollback, tool use, skills, and project memory (TERAX.md), BYOK across cloud and local providers.
- **Trust by default.** Every OS-touching operation crosses one audited Rust boundary; secrets live in the OS keychain; no telemetry.
- **Featherweight.** ~7-8 MB app, no bundled runtimes, unused features cost zero.

The SOTA version of this idea is the codebase where those four statements are true without exceptions. Today each has exactly one major exception, listed in section 3.

## 2. Where the codebase stands (delta since the 2026-06-10 review)

### 2.1 What shipped (commits 8560add..3d4fcff, +22.5k lines, 220 files)

**The webview-native Pi agent reached sidecar parity and became the default** (`USE_WEBVIEW_AGENT = true`, `pi-session.ts:545`). This is the decisive architectural event. The runtime now works like this:

- The agent loop runs in the webview on `@earendil-works/pi-agent-core` (`Agent`), with `pi-ai` streaming. The older SOTA doc's `createAgentSession` plan was abandoned in favor of bringing the hand-rolled path to parity. That pivot is accepted (see D1).
- Tools (`read_file`, `list_directory`, `grep`, `glob`, `write_file`, `edit_file`, `bash_run`, `ask_question`, plus discovered MCP tools) execute through standard Tauri commands via `pi-tools.ts`.
- Approval policies come from Rust (`pi_capability_manifest`, manifest is the single authority) but are enforced by an `approvalGate` wrapper in the webview (`pi-session.ts:425-449`), with a 5-minute auto-deny timeout.
- All LLM traffic is proxied through the SSRF-guarded Rust client (`ai_http_stream` / `ai_http_request`) via a ref-counted global fetch override (`pi-http.ts`).
- Persistence is two-file via new Rust commands: `pi_store_record_session/events` (capped 500-event log in `pi-sessions.json`) and `pi_store_record/load/delete_transcript` (canonical `AgentMessage[]`, atomic temp-file-rename writes, FNV-1a-hashed filenames).
- Fork, rollback, archive/restore, interactive questions, model switching on resume: all working in the webview path.

**Also landed:** capability framework applied to `fs_write_file`, `fs_delete`, `shell_run_command`, workflow file/http operations; MCP client refactor (connections split, OAuth rewrite, config store); 7 Rust integration test files (~4k LOC); idle-shutdown panic fix (`tokio::spawn` to `tauri::async_runtime::spawn`, with regression test); workflow gained schedule/webhook triggers, subgraphs, version snapshots, validation/docs panels; sidecar bundle shrunk; app shell split started (`AppSidebars.tsx`, `AppWorkspaceSurface.tsx`).

**Uncommitted:** a voice/TTS module (`src-tauri/src/modules/voice/`, Cartesia API + macOS AVSpeech via rodio, gated behind the `openclicky` feature) and a read-aloud action in `PiTranscript.tsx`.

### 2.3 What the pivot regressed, and what remains open

**The new #1 issue: approval enforcement moved to the wrong side of the trust boundary.** In the sidecar design, Rust verified every tool execution (`execute_verified_native_tool_with_policy`). In the webview design, Rust executes `fs_write_file` / shell commands trusting that the webview gate already ran. The policy lives in Rust but the enforcement lives in compromisable territory: a prompt-injected model, a bug in the gate wrapper, or a supply-chain compromise of any npm dependency in the webview can invoke mutating IPC directly, with no Rust-side re-check. The audit log likewise only sees what individual commands self-report.

**Audit ledger holes** (Rust): `pty_open/write/close` (interactive shell, the highest-risk surface) record nothing; `mcp_call_tool` records nothing; `tts_speak` makes an unaudited external network call with no timeout and unbounded text; `pi_store_record_transcript` accepts unvalidated, unbounded JSON (disk DoS, silent corruption on resume); session-history writes are non-atomic (truncation on crash).

**Dead weight still aboard:** the sidecar remains bundled (83 MB `pi-host/dist` + 91 MB `node/dist` of resources) and rebuilt on every `tauri build`, despite being a non-default fallback; the legacy `ai/` stack is untouched at 13,092 LOC; workflow carries dead code (`videoThumbnail.ts`, 168 LOC unused) and an unwired `webviewAgentExecution.ts`.

**Frontend debt unchanged from yesterday:** `App.tsx` still 934 LOC with 18 hooks and ref-map plumbing; `PiTranscript.tsx` (now 1,050+ LOC) still renders every event unvirtualized, which matters more now that pi is the flagship surface; `PiPanel` still drives 20+ reads through the custom `PiControllerProvider`; `chatStore` fan-out persists (moot once `ai/` folds).

**Workflow approval conflation:** approving a workflow node hardcodes `approved: true` into the Pi session policy (`nativeAgentExecution.ts:118-130`), so one click authorizes every tool the agent step may use, with no per-tool gating and no schema version migration for saved workflows (`WORKFLOW_DOCUMENT_VERSION = 1`, no migration code).

**Infra gates still open (re-verified today):** Biome linter disabled (`biome.json`), `pnpm audit` non-blocking (`ci.yml:29-30`), no bundle-size gate, no frontend coverage tracking, no e2e suite, zero tests in editor/git-history/theme.

## 3. The SOTA target architecture

One sentence per product-thesis exception, then the decisions that remove it:

1. *Terminal first*, except the app ships a 174 MB Node sidecar it no longer uses. Remove it (D2).
2. *Agent as a primitive*, except there are still two agent stacks (pi + legacy ai). Converge (D3).
3. *Trust by default*, except tool approval is enforced in the webview. Move enforcement into Rust (D1).
4. *Featherweight*, except dead fallbacks, dead code, and an unguarded TTS path ride along. Delete and harden (D2, D5).

### D1. Trust boundary doctrine: the webview proposes, Rust disposes

The single most important change. Options considered:

- **A. Status quo** (webview-only gate). Rejected: loses defense-in-depth precisely where the product promises it. The threat model for an agentic terminal includes prompt injection by construction; the enforcement point must sit below the layer the model can influence.
- **B. Move the whole agent loop into Rust.** Rejected: months of rewrite, loses the pi SDK, no incremental path.
- **C. Rust-side approval verification via grants.** Chosen. Mechanics:
  - When the user approves a tool call, the frontend calls a new `pi_approval_grant` command recording `(sessionId, toolCallId, toolName, argsHash)` in Rust state; denials are recorded too. This resurrects the sidecar-era `native_tool_approvals` machinery (`native_tools.rs:106-137`), which already exists and is tested.
  - Agent-initiated mutating calls (`fs_write_file`, `fs_create_*`, `fs_rename`, `fs_delete`, `shell_session_run`, `shell_bg_spawn`, `mcp_call_tool`) accept an optional `agentGrant: { sessionId, toolCallId }`. When present, Rust verifies the grant against policy (Auto passes, Ask requires a recorded grant, Deny blocks) and consumes it (single-use) before executing, then writes the audit entry.
  - Calls without a grant from the pi tool layer are rejected; direct user-driven UI calls (editor save, explorer rename) are unaffected because they do not originate from the agent bridge.
  - The webview gate stays as UX (it produces the approval card); it just stops being the security boundary.
  - The existing `check-pi-approval-boundary.mjs` string check is replaced by a behavioral test: deny a grant, assert no file written; approve, assert audit entry exists.

### D2. One runtime: delete the sidecar, the Node runtime, and the supervisor

The webview agent is at parity and default; the fallback now costs 174 MB of resources, sidecar rebuild time in every build, and a second code path in `pi-session-backend.ts`. Options: keep as fallback one more release (rejected: nothing routes to it and it will rot), or delete now. Chosen: delete now, in one commit series: `sidecars/`, `build-node-runtime.mjs`, `build-pi-host-bundle.mjs`, `smoke-pi-host-bundle.mjs`, tauri resources entries, `beforeBuildCommand` sidecar step, `pi/host.rs` + `host/bridge.rs` + `host/protocol.rs` supervisor (keep `native_tools.rs` for D1, `store.rs`, `state.rs`), the `sidecarBackend` branch, and the `USE_WEBVIEW_AGENT` flag itself. The git history preserves it if ever needed.

### D3. One agent UX: fold legacy `ai/` onto pi sessions

Unchanged from yesterday's plan in intent, now easier: provider/model config is already unified, and the pi runtime is proven. Migrate the docked composer, mini window, and status-bar controls to pi sessions; provide one-time import of `terax-ai-sessions.json`; then delete `ai/lib/agent.ts`, `ai/tools/`, the chatStore agent loop, and the AI-SDK chat transport (~10k of the 13k LOC). The AI SDK provider packages stay (they power `ai_http_*` model calls through `ai/config.ts`). Slice order: composer, mini window, sessions list, bridges.

### D4. Workflow: two-level approval and a stable schema

- Split node-approval ("run this agent step") from tool-approval ("may this step write files"). The Pi session created for a workflow step gets a policy snapshot, not `approved: true`; Ask-level tools either pause the run for an inbox approval or require the user to pre-grant named tools on the node ("this step may: write_file, bash_run"), which becomes part of the saved document and is shown at run time. Pre-granting is the default UX; it keeps unattended runs possible while making authority explicit and auditable. Wire `webviewAgentExecution.ts` (currently dead) as the execution path so workflow agents run on the same runtime as chat (one runtime everywhere, including D1 grants).
- Add `migrateWorkflowDocument` with a version enum before any further schema change; round-trip tests.
- Delete `videoThumbnail.ts`; split `WorkflowCanvas.tsx` (1,053 LOC) into memoized node/edge/panel children.

### D5. Voice as a first-class capability, or not at all

The uncommitted TTS work is product-appropriate (read agent replies aloud) but currently violates the boundary rules. Before commit: route the Cartesia call through the audited net layer (timeout, response-size cap, capability audit entry `app.tts`), cap input text (~4 KB), keep the macOS-native path as default when available, and decide the feature gate honestly (it currently hides behind `openclicky`, which is unrelated; give it its own `voice` feature or none). Streaming TTS and STT can come later; the boundary comes first.

### D6. Frontend architecture (carried forward, re-prioritized)

1. **Virtualize `PiTranscript`** with the existing `@tanstack/react-virtual` dependency, preserving stick-to-bottom. Now the top frontend item: pi is the flagship surface and sessions are long.
2. **Replace `PiControllerProvider` with a zustand store** with narrow selectors; split `PiPanel` (1,008 LOC) into list/transcript/composer children.
3. **Finish the App.tsx split** the shell refactor started: surface contexts (code, agent, layout) instead of 17 ref maps and 18 callbacks; collapse the 18 `useApp*` hooks into 4 coordinators; keep the two documented invariants (unconditional `AiComposerProvider` mount, never-unmount tabs) locked by a regression test. Target under 400 LOC.
4. chatStore selector work becomes mostly moot via D3; apply narrow selectors to whatever survives.

### D7. Backend hardening checklist

- PTY commands wrapped in capability audit (policy Auto, but recorded): the ledger must be complete to be a ledger.
- `mcp_call_tool` audited with tool name and server id.
- `pi_store_record_transcript`: parse-validate JSON, cap size (10 MB), reject otherwise; make session-history writes atomic (same temp-rename pattern as transcripts).
- Expand the secret deny-list (`~/.aws/credentials` file, `~/.docker/config.json`, `~/.config/gcloud/`, `~/.kube/config`).
- Remove remaining vestiges (`sync.rs`, `proc.rs` if unused) and the `#![allow(dead_code)]` HTTP MCP file-level escape.

### D8. Quality gates (unchanged from yesterday, still all open)

Enable Biome linter in CI; make `pnpm audit` blocking for high/critical; bundle-size budget gate after `vite build`; frontend coverage with floors; e2e golden flows via WebdriverIO + tauri-driver on Linux (terminal cwd, file edit/save, git stage/commit, pi prompt + approval round-trip with a mocked provider); tests for editor/git-history/theme; shellcheck on PTY init scripts; component-size warning at 600 LOC.

**2026-06-11 (Phase C/D re-assessment against the code).** After reading the actual implementations:
- **`PiControllerProvider`→zustand is NOT worth doing.** The earlier audit assumed it caused wide re-renders. It does not: each `usePiControllerState(key, initial)` is an independent `useState`; the ref-held store is only a *persistence* layer (it retains values across panel remounts), not a shared-subscription store. The `TranscriptItem` memoization (done) captured the safe, high-value part (no more all-rows re-render per streaming token).

**2026-06-12 (Phase D: App.tsx split + transcript virtualization).** Two of the deferred frontend items landed now that the e2e net exists.
- **App.tsx split.** Pulled three self-contained JSX clusters out of `App.tsx` (934 → 888 lines) into named presentational components: `AppBridges` (notification/event bridges + toast portal, composer-gated pair included), `AppOverlays` (command palette, shortcuts, new-file, updater, close dialogs), and `AppComposerDock` (the workspace surface + docked AI input bar, preserving the never-unmount/`inert` collapse). Pure render split - the coordinator hook sequence (which must stay in the component body, and whose ordering is load-bearing) is untouched, as are the `AiComposerProvider`/`PiControllerProvider` trees. The remaining ~470 lines are inherent: they are the coordinator wiring that threads each `useApp*` output into the next, which cannot be extracted without breaking hook order.
- **Transcript virtualization - chose native `content-visibility` over `@tanstack/react-virtual`.** `react-virtual` would have to co-own the scroll container with `use-stick-to-bottom`'s spring-based auto-follow; the two fight over `scrollTop` during streaming, which is exactly the anchor risk this item was deferred for and cannot be tuned without interactive testing. Instead, a shared `LazyRow` (`src/components/lazy-row.tsx`) wraps each non-streaming row in `content-visibility: auto` + `contain-intrinsic-size: auto 64px`, so off-screen rows skip layout/paint but stay in the DOM - the scroll height stays correct and StickToBottom is undisturbed. This let me **remove the hard 160/80 render caps** in `PiTranscript` and `AiChat` (which silently dropped older messages); the full history is now reachable, and unsupported WebViews simply ignore the properties and render normally. The streaming row stays `eager` (no containment, since it is on-screen and changes per token). The old "windows large transcripts" test now asserts the full history renders with `content-visibility:auto`. 844 frontend tests green, build clean, bundle 1786.7 KB (budget 2050.8 KB).

**2026-06-12 (Phase C: convergence design + first down-payment).** Mapped the real dependency graph before touching anything and it reshaped the work: `ai/` (~13.5k LOC) is consumed by ~55 files and splits into three layers - **shared infra** (keyring, model discovery, transport, security, the `AiComposerProvider`) that pi and settings already depend on; a **shared agent runtime** (`chatStore`, `tools/*`, `agents/*`) that workflow, model-compare, source-control, and agents all use; and only then the **chat surface** (`AiChat`/`AiInputBar`/`AiMiniWindow`/plan+todo) that genuinely duplicates pi's own surface (`PiComposer`/`PiTranscript`/`PiFloatingWindow`). A blind "delete `ai/`" would break settings, workflow, model-compare, source-control, git-history, editor autocomplete, and agents - and the converged surface cannot be regression-tested yet because the chat path needs a live BYOK provider (keys + network), which the e2e harness deliberately excludes. So Phase C is a staged sequence behind a stable public surface, written up in **`docs/phase-c-convergence-plan.md`**: Stage 0 stands up a deterministic mock provider + e2e (the precondition for changing the surface), Stage 1 unifies the run-status contract underneath the shared consumers, Stage 2 lets the pi runtime back the docked quick-ask composer behind a flag, Stage 3 retires the duplicate surface pieces (mini window, plan-diff, todo strip, transcript) once pi covers them, and Stage 4 collapses/renames the runtime. **First down-payment already landed:** the shared `LazyRow` (above) is exactly the convergence pattern - extract the genuinely-shared primitive, point both surfaces at it, retire the legacy side once pi covers the affordance. The deeper stages are intentionally not forced here: each needs the Stage 0 verification scaffold first, and landing a 10k-line surface migration blind into five+ dependent features without a test path would regress a core UX.
